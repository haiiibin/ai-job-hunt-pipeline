# JD 评分标准(A-F Scoring Rubric)

收到 JD 后,先输出评分表,再决定是否进入简历生成。/hunt 用它做初评排序,/apply 用它做正式评分。

## 维度与权重

| 维度 | 权重 | 说明 |
|---|---|---|
| A:简历匹配度 | 30% | 技能、经历、证书与JD关键词重合程度 |
| B:职业路径契合 | 25% | 岗位方向是否符合你的目标方向(在 hunt.md 用户配置中定义) |
| C:薪资水平 | 20% | 以你当前/期望薪资为基准对标(5=明显高于基准;1=低于基准) |
| D:公司与文化信号 | 15% | 成长空间、稳定性、团队结构、远程政策 |
| E:红旗扣分 | 10% | 职责描述矛盾、幽灵岗位迹象、过度要求 |
| F:综合评分 | -- | 加权平均(1-5分) |

## 阈值(可自调)

- F ≥ 4.5:优先投
- 4.0-4.4:正常投
- 3.5-3.9:需要额外理由才投
- < 3.5:不建议投
- **F ≤ 2.0:停下,明确告知不建议申请**,问用户是否仍要继续;F > 2.0 直接进简历生成,不询问

## 岗位真实性三问

逐岗核对,任一不满足标 ⚠️:

1. 发布 < 30 天?
2. 有明确申请入口?
3. 要求合理(无过度要求堆叠)?

Boss 渠道加看 HR 活跃度:活跃文本含"月/年"字样(如"3月内活跃""半年前活跃")= 不活跃,标 ⚠️幽灵嫌疑。

## 评分纪律(反猜测约束)

- 评分前先过 `pipeline/filter_rules.yaml` 硬规则粗筛:hard_veto 命中不入库、flag 命中强制二档、safe_phrases 防误杀("非外包"不算命中"外包"),不给垃圾岗浪费评分
- **E 红旗只认 JD 原文明确出现的红线词句**(外包、单休、职责矛盾等);严禁基于"可能高压""竞争激烈""估计要加班"等推测扣分。信息不足按中性处理
- **JD 已标明薪资/城市/工时的,C、D 维度必须按标明数据打分**,禁止写"未披露"再给中性分
- **每个维度附一句话依据**,须引用 JD 或简历原文,不空泛

## 幽灵信号与词库衔接

`pipeline/filter_rules.yaml` 的 `ghost_signals` 节列出幽灵岗位信号(发布>30天无更新 / 无申请入口 / Boss HR 活跃文本含"月/年" / 在库 status=new 超30天无变化)。命中不拦截,计入 E 维度扣分,报告标 ⚠️幽灵嫌疑。

硬规则词库三层优先级:safe_phrases(白名单,最高)> hard_veto(不入库)> flag(强制二档+标注)。词条按自身求职方向增删(见该文件头部注释)。

## 输出格式

```markdown
| 维度 | 得分 | 依据(引用 JD/简历原文) |
|---|---|---|
| A 简历匹配 | x.x | ... |
| B 路径契合 | x.x | ... |
| C 薪资 | x.x | ... |
| D 公司信号 | x.x | ... |
| E 红旗 | x.x | ... |
| **F 综合** | **x.x** | 加权 |

真实性三问:发布<30天 ✓/⚠️ | 申请入口 ✓/⚠️ | 要求合理 ✓/⚠️
```
