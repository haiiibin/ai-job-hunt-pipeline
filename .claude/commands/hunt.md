# /hunt -- 秋招岗位雷达(收集 → 核验 → 初评 → 报告)

把"发现岗位"这一环自动化:扫四类渠道,产出新增/变化岗位的两档清单报告。**报告为止**:用户勾选后逐岗走 `/apply`(/apply 内有名额门与提交回写)。

`$ARGUMENTS` 渠道过滤:
- 空 = 全渠道(官网监控清单+扩展名单、牛客汇总、智联前程社招、Boss只读)
- `官网` = 快扫,只扫监控清单
- `牛客` / `社招` / `boss` = 对应单渠道

**铁律(全程任何阶段):绝不提交任何表单;绝不通过 Boss 或任何平台发消息/打招呼;登录一律用户手动;绝不点"解析并覆盖"。**

## 用户配置(使用前编辑本节)

以下拍板项决定扫描范围与打分口径,首次使用前按自身情况改写:

| 配置项 | 本仓库默认值(示例) | 说明 |
|---|---|---|
| 目标城市 | 北京优先;同公司无北京对口岗时才收上海岗(报告标"沪·无京替代");其它城市不入库 | 改成你的城市优先级 |
| 毕业时间窗 | YYYY.MM(例: 2024.06) | 逐岗核对 JD 原文的毕业窗口是否包含此年月 |
| 届数身份 | 26/27届(留学生口径) | 校招批次判定用 |
| 薪资基准 | 不设硬底线;仅进 C 维度评分,基准 = 你当前/期望月薪 | 高于基准加分,不因低薪卡入库 |
| 目标方向 | 数据分析/商业分析/数据产品/战略/运营/投研,四方向均等不加权 | 改成你的方向清单 |
| 监控清单 | 见下方渠道①的 5 家表(按自身目标替换) | 每轮必扫的公司官网 |

## Phase 0 -- 读状态

1. Read `pipeline/jobs.yaml`(去重库+名额账本,用户自建,已 gitignore)。文件缺失则按下述 schema 初始化空账本,不凭空造记录:

   ```yaml
   # 机器状态权威: 去重库 + 大厂名额账本。由 /hunt 与 /apply 维护。
   meta:
     created:          # YYYY-MM-DD
     last_scan:        # YYYY-MM-DD, /hunt 每轮更新
     filter_rules: pipeline/filter_rules.yaml

   quotas: {}          # 有投递名额上限的公司才登记
   # 示例条目:
   #   某公司: {limit: 2, used: 0, note: ""}

   jobs: []            # 岗位记录数组
   # 示例条目字段:
   #   - id:           # 去重键: 公司英文/拼音 + jobId
   #     company:
   #     title:
   #     url:
   #     location:
   #     batch:        # 如 2026届校园招聘 / 社招
   #     channel:      # 官网 / 智联 / 前程 / boss / 牛客线索
   #     first_seen:   # YYYY-MM-DD
   #     last_seen:    # YYYY-MM-DD
   #     eligibility: {verdict: , reason: ""}   # verdict: 一档/二档/待登录核
   #     score_f:      # F 初评(一位小数)
   #     status:       # new / candidate / applied / rejected / expired
   ```

2. Read `pipeline/watchlist.md`(监控清单+资格速记+已投递记录,用户自建,已 gitignore;结构见 README)。
3. 时间锚定:秋招窗口约 7 月初至 10 月底(16 周),算今天的位置,报告开头画进度条。

## Phase 1 -- 渠道扫描(串行)

通用规则:
- Playwright MCP 驱动真实 Chrome(browser_navigate / browser_snapshot);快照尽量带 target 选择器抓局部,省 token。
- 每渠道独立:失败或单渠道约 15 分钟无进展 → 记"本轮未扫成+原因",继续下一渠道,不拖垮全轮。
- 登录墙:暂停请用户登录(手机验证码,账号密码不经手);用户不在场 → 跳过该渠道并在报告标注。
- 反爬/页面改版读不到岗位列表 → 降级 WebSearch 该公司招聘动态,报告标"需人工看一眼"。
- 节流(借 boss-agent-cli 思路):同渠道内翻页/点详情间隔随机 2-5 秒,连续动作不密集突发;触发滑块/安全验证 → 暂停请用户手动完成后继续,绝不自动过验证码。

### 渠道① 大厂官网直扫(每轮必扫)

驱动清单 = 用户配置里的监控清单。以下 5 家为示例(2026-07 实测入口,按自身目标替换):

| 公司 | 入口 |
|---|---|
| 腾讯 | join.qq.com/post.html(勾"2026校园招聘";留学生看 Pre 留学生) |
| 字节 | jobs.bytedance.com/campus(左侧按招聘项目过滤,URL 带 project= 才生效) |
| 阿里 | campus-talent.alibaba.com/campus/index(统一校招入口)+ talent.alibaba.com |
| 美团 | campus.meituan.com |
| 京东 | campus.jd.com(新星计划补录 + 京东国际/海外专场) |

全渠道模式追加扩展名单(互联网示例):小红书 job.xiaohongshu.com、快手 campus.kuaishou.cn、网易 campus.163.com、百度 talent.baidu.com、拼多多(官网校招入口)。

行业扩展名单(示例,按自身方向替换,四类可全开可裁剪):
- 金融机构:头部券商/公募基金/银行金融科技子公司
- 外企科技在华:微软/苹果/亚马逊/特斯拉中国
- 消费快消:宝洁/联合利华/蒙牛等管培与数字化岗
- 独角兽/新贵:米哈游/得物/SHEIN/宁德时代

行业扩展**以渠道②(牛客/汇总站)+ WebSearch 为发现层**,信号命中(开闸/在招对口岗)才去该公司官网直扫核实,不逐家全扫(控制单轮时长)。

每家三查:
1. 留学生/海外专场是否开闸(常"毕业2年内"口径,7-9月陆续开);
2. 在招对口业务岗(方向清单见用户配置);**地点规则见用户配置**;
3. 该岗毕业窗口**原文**是否含用户配置的毕业年月(逐岗判,不用公司总口径)。

岗位详情需登录才能见原文 → 该岗 `verdict: 待登录核`,不用摘要糊弄。

### 渠道② 牛客 + 汇总站(发现层,免登录)

- nowcoder.com(大厂校招最全)、campus2026.top、qiuzhifangzhou.com/campus。
- 只当信号源:新开专场/新批次/某大厂业务岗开闸。命中 → 回该公司官网核实后才入库;**绝不凭汇总站信息定资格**。

### 渠道③ 智联 + 前程无忧(社招口径)

- 搜索矩阵:{方向清单(见用户配置)} × {目标城市(见用户配置)}。
- 经验过滤:要求 ≤2 年的收;"1-3年"临界岗收进二档;外包/派遣岗标注(flag 词库),由用户判断。
- 薪资不设硬底线(见用户配置):薪资只进 C 维度评分,高于基准即加分,不因低薪卡入库。
- 智联社招走 i.zhaopin.com(与校园站两套系统);登录态持久于 `.playwright-profile`(仓库根,已 gitignore)。

### 渠道④ Boss 直聘只读扫描

- 只搜索、只浏览。**任何操作都不得触发消息、打招呼、投递。**
- 命中岗:优先找同岗的官网/表单制渠道入口;确实只有 Boss 入口 → `channel: boss` 入库,报告注明"需起草消息由用户亲手发"。

## Phase 2 -- 资格核验(逐岗)

### Step 0 -- 硬规则粗筛(零成本,先于一切人工判断)

Read `pipeline/filter_rules.yaml`,对每个候选岗按优先级过三层(借 Auto-JobHunter 双漏斗:硬规则先行,不给垃圾岗浪费核验与评分):

1. **safe_phrases 先行**:标题/JD 命中白名单短语时,短语内包含的词不算命中(如"非外包"不触发"外包");
2. **blacklist.companies**:命中公司 → 直接跳过,记入报告"过滤明细";
3. **hard_veto**(title 层查标题,jd 层查正文)→ 不入库,记入"过滤明细"(公司/岗位/命中词,可追溯);
4. **flag** → 不淘汰,强制归二档并标注命中词,由用户自判。

### 逐岗核验

- **批次**:校招岗只收 秋招/秋招提前批/留学生专场;春招与已截止剔除。社招岗无批次,按经验要求过。
- **毕业窗口**:按"这条岗"的 JD 原文判用户配置的毕业年月是否在窗;公司总口径或其它批次摘要不可套用。
- **身份口径**:统招含留学生(留服认证视同统招);海外工作无国内社保 = 应届身份不破;"无全职工作经验"条不预先卡人,归二档。
- **两档呈现,不预先淘汰**:一档 = 资格+对口都确定;二档 = 不确定但匹配(资格临界/需试/方向沾边),列给用户自己判断。
- 读不到 JD 原文 = `待登录核`。

## Phase 3 -- F 初评

- 按 `docs/scoring-rubric.md` 的 A-E 维度对标题+可见 JD 快评,加权得 F 初评(一位小数)记入 `score_f`。
- B 维度口径:用户配置的目标方向均等对待,不做方向排序加权,只看单岗与简历的实际匹配度。
- **评分纪律(反猜测约束,全文见 docs/scoring-rubric.md)**:E 红旗只认 JD 原文明确红线,禁止"可能加班/竞争激烈"类推测扣分;信息缺失按中性,不猜;JD 已标薪资/城市的必须按标明数据打,禁写"未披露"。
- **幽灵信号**(filter_rules.yaml `ghost_signals`):发布>30天无更新 / 无申请入口 / Boss渠道HR活跃文本含"月/年" / 在库 status=new 超30天无变化 → E 维度扣分,报告该岗标 ⚠️幽灵嫌疑。
- 这是排序用初评;正式评分(全 JD + 真实性三问)在 /apply Phase 2 做。

## Phase 4 -- 写状态

- 去重键 `id` = 公司英文/拼音 + jobId;无 jobId 用规范化 URL(剥离 query 追踪参数)。
- 新岗 → `status: new`;重见岗 → 仅更新 `last_seen`;上轮 new/candidate 本轮下线/截止 → `expired`。
- **`quotas.used` 只由 /apply 回写,本命令绝不改。**
- `meta.last_scan` 更新为今天。
- 同步 `pipeline/watchlist.md`:
  - "候选池"节:按 jobs.yaml 的 new/candidate 重写(两档分组);
  - "监控清单"表:更新各家开闸状态与检查日期;
  - 顶部加或更新一行"最近扫描:YYYY-MM-DD(/hunt)";
  - **"已投递记录"表绝不动**(由 /apply 或用户维护)。

```markdown
## 新增与变化

| 公司 | 岗位 | 地点 | 批次 | 资格 | F初评 | 名额 | 链接 |
### 二档 -- 可试
(同表结构)

## 名额警示
- {公司} X/Y 已满;{公司} X/Y(未面试可换投);...

## 渠道回执
- 官网 N/N 成功;牛客 成功;智联 跳过(登录失效);Boss ...

## 过期清理

| 公司 | 岗位 | 拦截层 | 命中词 |

## 下一步

```

## Rules

- 报告只含新增/状态变化;完整候选清单看 watchlist.md 或 jobs.yaml。
- 全程输出禁用 em dash。
- 资格/信息不确定 → 标"待核",不猜。
- jobs.yaml 是机器状态权威,与 watchlist.md 冲突以它为准。
