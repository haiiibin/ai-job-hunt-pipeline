# /apply <JD> -- 网申全链路（评分 → 定制简历 → 填表 → 待提交）

把一个岗位从 JD 链接走到"表单填好、停在提交前"。面向中国官网/校招网申。

`$ARGUMENTS` = JD 链接（优先）或粘贴的 JD 文本。缺失则问用户要。

**铁律（贯穿全程，任何阶段都不得违反）：**
- **绝不自动提交**：填完截图，由用户亲手点提交。
- **绝不点"解析并覆盖"**：会用 PDF 覆盖掉已填字段。出现即忽略/关 ×。
- **登录由用户手动**：手机号+短信或邮箱密码，账号密码我不经手。
- **简历每次按本 JD 重新生成**，不沿用旧档案 PDF。

---

## Phase 0 -- 读规则（必做，按顺序）

以下路径均相对仓库根：

1. `CLAUDE.md` 已在上下文：复习工作流约定与输出规范。
2. Read `docs/cn-workflow.md`（中文简历生成 + 格式锁定，完整读）。
3. Read `docs/webapp-autofill.md`（浏览器填表流程 + 坑）。
4. Read `pipeline/profile.yaml`（个人网申字段档案：证件号/微信/邮箱/手机/信息来源渠道等固定值。用户自建，已 gitignore，字段清单见 README）。

## Phase 1 -- 抓取 JD

- 链接是 JS 渲染站（字节/腾讯/阿里招聘等）→ **WebFetch 通常只拿到导航壳**。直接用 `browser_navigate` + `browser_snapshot` 读正文 JD。
- 静态页或用户已粘贴文本 → 直接用。
- 提取：岗位名、岗位类型（data / finance / consulting / marketing / ops / product）、硬性要求（应届/学历/语言/地点）、关键词。

## Phase 2 -- JD 评分（A-F，见 docs/scoring-rubric.md）

输出评分表（A 匹配 / B 路径 / C 薪资 / D 公司 / E 红旗 → F 加权）+ 岗位真实性三问（发布<30天 / 有申请入口 / 要求合理）。

- **F ≤ 2.0**：停下，告知不建议申请，问用户是否仍要继续。
- **F > 2.0**：直接进 Phase 3，不询问。
- **硬性资格不符**（如校招岗要应届、用户已毕业在职）：标出来，但**由用户判断**是否投。可建议改投社招同类岗。

## Phase 2.5 -- 名额门(大厂投递名额检查)

1. Read `pipeline/jobs.yaml` 的 `quotas`。目标公司不在表中或无名额上限 → 直接进 Phase 3。
2. **名额已满(used >= limit)→ 停**:告知用户,建议换渠道(社招/留学生专场)或等名额释放(部分公司未进面试前可换投)。
3. **只剩最后 1 个名额(used == limit - 1)→ 强制通览**:部分大厂限投 1-3 岗,名额金贵。先用浏览器通览该公司全部在招对口岗,列对比表,请用户确认当前岗是最优选后才继续:最后一个名额必须挑最准的。

## Phase 3 -- 定制中文简历（按 docs/cn-workflow.md）

1. 读 `resume-data/_index.yaml` → 按 JD 类型/tags 选经历与项目（教育+格式首次填好后锁定，见 `templates/cn_skeleton.html`）。
2. 动态开关：按 JD 类型开关可选内容模块（页眉 GitHub、仅特定岗位类型才放的经历、第三章节标题、技能行排序，规则在 cn-workflow 中定义）。
3. 输出目录 `output/[公司]_[岗位简称]/`，文件名 `[你的姓名]_CV_[岗位简称].html` / `.pdf`（**不带 CN_**）。
4. **以资深 HR 视角撰写 bullet**：从 `resume-data/` 真实细节展开，每条**约 1.5 行（55-72 字含标签）、不超过两行**，长度统一、不空泛。
5. 用 `compile_cv.py` 生成 PDF，然后**量化验满页**（cn-workflow 第八步）：PyMuPDF 测 `fill% ≥ 93%` 且 `pages==1`；不足则置顶经历 2→3 条 bullet、写满 1.5-2 行，重编直到达标。
6. **AUTHENTICITY**：只用真实经历，指标据实，不编造。

先把简历给用户看一眼（PDF 路径 + 截图核对满页），等用户认可再进 Phase 4。简历有疑问就在这里改完再填表。

## Phase 4 -- 浏览器填表（按 docs/webapp-autofill.md）

1. `browser_navigate` 打开 JD → 点"投递/申请" → 登录页。
2. **暂停，请用户登录**（同一公司 session 持久，下次免登）。用户确认登录后继续。
3. `browser_snapshot` 抓表单。**警惕门户自动带出的旧档案**（旧简历/过时经历）：逐项核对覆盖，不信旧值。
4. 填：基本信息、教育（锁定内容）、工作/实习/项目（来自本次简历）、自我评价、语言、社交、了解渠道。长表单用 `browser_fill_form` 批量。
   - 自定义下拉：点开 → portal 里用 `text=选项文字` 点选（`browser_select_option` 会报错）。
   - 当前在职结束日期：日期面板底部选"至今"。
   - 渠道两级（字节类）：主选"招聘官方渠道/账号"，追问选"校园招聘官网"。
5. **上传简历**：先 `cp` PDF 到 `.playwright-mcp/`（仓库根，已 gitignore），再 `browser_file_upload`（allowed-root 限制）。显示名 = 副本名 `[你的姓名]_CV_[岗位简称].pdf`。
6. **上传后若弹"解析并覆盖" → 忽略/关闭，绝不点。**
7. 每次 DOM 变动（增删/上传）后重新快照，ref 会失效。

## Phase 5 -- 截图待提交（终点）

1. `browser_take_screenshot` 全表 → 给用户核对。
2. 列出关键字段回执（姓名/手机/邮箱/教育/各段经历/渠道/简历附件名）。
3. 明确告诉用户：**已停在提交前，请你核对后亲手点"提交"。**
4. **不调用任何提交动作。**

## Phase 6 -- 提交回写(仅当用户确认"已亲手提交"后执行)

1. 更新 `pipeline/jobs.yaml`:该岗 `status: applied`(不在库中则新建记录);公司在 `quotas` 中则 `used` +1。
2. 更新 `pipeline/watchlist.md` "已投递记录"表:加一行(日期 / 公司·岗位 / 渠道 / F分 / 简历路径 / 状态)。
3. 用户未确认提交 → 什么都不写,岗位保持 `candidate`(待提交)。

## 输出骨架

```markdown
# 📮 /apply -- {公司} · {岗位}

## 1. JD 评分
{A-F 表 + F 分 + 真实性三问 + 是否建议投}

## 2. 定制简历
- 选用经历：{...}  选用项目：{...}  开关：{GitHub? 可选经历? 第三章节标题}
- 文件：`output/{公司}_{岗位简称}/[你的姓名]_CV_{岗位简称}.pdf`（单页✓）

## 3. 填表状态
- 登录：用户已手动完成
- 已填字段：{清单}
- 简历附件：{显示名}
- ⚠️ 未点"解析并覆盖"｜⚠️ 未提交

## 4. 截图
{browser_take_screenshot}

## 5. 待你操作
请核对截图，确认无误后**亲手点提交**。
```

## Rules

- **绝不自动提交、绝不点"解析并覆盖"、登录用户手动**：三条铁律，全程生效。
- 评分用真实 JD 内容，不臆造薪资/要求。
- 简历内容只来自 `resume-data/`，据实量化，不编造。
- 表单文本框不必和 PDF 逐字一致（PDF 是主材料）。
- 任一字段不确定 → 问用户，不猜。
